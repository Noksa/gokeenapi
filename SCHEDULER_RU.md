# 🕐 Scheduler - Автоматизированное выполнение задач

Планировщик - это мощная функция, которая позволяет автоматизировать управление роутером, выполняя задачи с заданными интервалами или в фиксированное время. Идеально подходит для автоматического обновления маршрутов и DNS записей без ручного вмешательства.

## Обзор

Планировщик работает как долгоживущий процесс, который непрерывно отслеживает настроенные задачи и выполняет их согласно расписанию. Он разработан для надежности, эффективности и простоты настройки.

**Сценарии использования:**
- Автоматическое обновление маршрутов из динамических источников каждые несколько часов
- Обновление DNS записей в определенное время ежедневно
- Выполнение задач обслуживания в непиковые часы
- Управление несколькими роутерами с помощью одного экземпляра планировщика

## Ключевые возможности

### ⏰ Гибкое планирование

- **По интервалу**: Запуск задач каждые N часов/минут (например, `"3h"`, `"30m"`)
- **По времени**: Запуск задач в определенное время (например, `["06:00", "12:00", "18:00"]`)

### 🔗 Цепочка команд

Последовательное выполнение нескольких команд в одной задаче:
```yaml
commands:
  - delete-routes  # Сначала удаляем старые маршруты
  - add-routes     # Затем добавляем новые
```

### 🔄 Механизм повтора

Автоматический повтор неудачных задач с настраиваемым количеством попыток и задержкой:
```yaml
retry: 3           # Повторить до 3 раз
retryDelay: "30s"  # Ждать 30 секунд между попытками
```

### 🎯 Поддержка нескольких роутеров

Управление несколькими роутерами одной задачей:
```yaml
configs:
  - /path/to/router1.yaml
  - /path/to/router2.yaml
  - /path/to/router3.yaml
```

### 📋 Последовательное выполнение

Задачи выполняются в очереди FIFO, что исключает конфликты при одновременном срабатывании нескольких задач.

## Быстрый старт

### 1. Создайте конфигурацию планировщика

Создайте файл `scheduler.yaml`:

```yaml
tasks:
  - name: "Обновление маршрутов каждые 3 часа"
    commands:
      - add-routes
    configs:
      - /path/to/router1.yaml
    interval: "3h"
```

### 2. Запустите планировщик

```bash
./gokeenapi scheduler --config scheduler.yaml
# Псевдонимы: schedule, sched
```

### 3. Мониторинг выполнения

Планировщик будет отображать:
- Расписание задач при запуске
- Прогресс выполнения
- Статус успеха/неудачи
- Попытки повтора (если настроено)

## Конфигурация

### Структура задачи

```yaml
tasks:
  - name: "Название задачи"          # Обязательно: Описательное название
    commands:                         # Обязательно: Список команд для выполнения
      - command1
      - command2
    configs:                          # Обязательно: Список конфигов роутеров
      - /path/to/config1.yaml
      - /path/to/config2.yaml
    interval: "3h"                    # Опционально: Интервал выполнения
    times:                            # Опционально: Время выполнения (24ч формат)
      - "06:00"
      - "12:00"
    retry: 3                          # Опционально: Количество попыток повтора (по умолчанию: 0)
    retryDelay: "30s"                 # Опционально: Задержка между попытками (по умолчанию: "1m")
    strategy: "parallel"              # Опционально: Стратегия выполнения (по умолчанию: "sequential")
```

### Описание полей

| Поле         | Тип    | Обязательно | Описание                                            |
|--------------|--------|-------------|-----------------------------------------------------|
| `name`       | string | Да          | Описательное название задачи                        |
| `commands`   | array  | Да          | Список команд gokeenapi для выполнения              |
| `configs`    | array  | Да          | Список путей к конфигурационным файлам роутеров     |
| `interval`   | string | Нет*        | Интервал выполнения (например, "30m", "1h", "3h")   |
| `times`      | array  | Нет*        | Время выполнения в формате HH:MM                    |
| `retry`      | int    | Нет         | Количество попыток повтора при ошибке (≥ 0)         |
| `retryDelay` | string | Нет         | Задержка между попытками (≥ 1s, по умолчанию: "1m") |
| `strategy`   | string | Нет         | Стратегия выполнения: "sequential" или "parallel"   |

*Должно быть указано либо `interval`, либо `times`, но не оба одновременно.

### Правила валидации

- `interval` должен быть ≥ 1 секунды
- `times` должны быть в 24-часовом формате (HH:MM)
- `retry` должен быть ≥ 0
- `retryDelay` должен быть ≥ 1 секунды
- `strategy` должна быть "sequential" или "parallel" (по умолчанию: "sequential")
- Нельзя использовать одновременно `interval` и `times` в одной задаче

## Режимы выполнения

### Стратегия выполнения

Управление тем, как команды выполняются для нескольких конфигураций роутеров:

#### Sequential (Последовательно, по умолчанию)

Команды выполняются для одного конфига за другим:

```yaml
- name: "Последовательное выполнение"
  commands:
    - add-routes
  configs:
    - /path/to/router1.yaml
    - /path/to/router2.yaml
    - /path/to/router3.yaml
  interval: "3h"
  strategy: "sequential"  # или не указывать (по умолчанию)
```

**Порядок выполнения:**
```
router1.yaml → router2.yaml → router3.yaml
```

**Использовать когда:**
- Роутеры используют общие ресурсы
- Важен порядок выполнения
- Предпочтителен консервативный подход

#### Parallel (Параллельно)

Команды выполняются одновременно для всех конфигов:

```yaml
- name: "Параллельное выполнение"
  commands:
    - add-routes
  configs:
    - /path/to/router1.yaml
    - /path/to/router2.yaml
    - /path/to/router3.yaml
  interval: "3h"
  strategy: "parallel"
```

**Порядок выполнения:**
```
router1.yaml ┐
router2.yaml ├─ Все выполняются одновременно
router3.yaml ┘
```

**Использовать когда:**
- Роутеры независимы друг от друга
- Требуется более быстрое выполнение
- Пропускная способность сети позволяет

### Выполнение по интервалу

Периодический запуск задач с фиксированным интервалом:

```yaml
- name: "Обновление маршрутов каждые 3 часа"
  commands:
    - add-routes
  configs:
    - /path/to/router.yaml
  interval: "3h"
```

**Поддерживаемые форматы:**
- `"30s"` - 30 секунд
- `"5m"` - 5 минут
- `"1h"` - 1 час
- `"2h30m"` - 2 часа 30 минут
- `"24h"` - 24 часа

**Поведение:**
- Первое выполнение происходит сразу при запуске планировщика
- Последующие выполнения происходят через указанный интервал
- Таймер сбрасывается после завершения каждого выполнения

### Выполнение по времени

Запуск задач в определенное время суток:

```yaml
- name: "Обновление DNS в фиксированное время"
  commands:
    - add-dns-records
  configs:
    - /path/to/router.yaml
  times:
    - "06:00"  # 6 утра
    - "12:00"  # 12 дня
    - "18:00"  # 6 вечера
```

**Поведение:**
- Выполняется в каждое указанное время
- Если все времена уже прошли сегодня, ждет до самого раннего времени завтра
- Использует 24-часовой формат (HH:MM)
- Автоматически учитывает часовой пояс

## Механизм повтора

Механизм повтора помогает справляться с временными сбоями, такими как проблемы с сетью или недоступность роутера.

### Конфигурация

```yaml
- name: "Обновление маршрутов с повтором"
  commands:
    - add-routes
  configs:
    - /path/to/router.yaml
  interval: "1h"
  retry: 3           # Всего попыток = 1 + 3 = 4
  retryDelay: "30s"  # Ждать 30 секунд между попытками
```

### Поведение

1. **Первая попытка**: Команда выполняется нормально
2. **При ошибке**: Если команда не выполнилась, ждем `retryDelay` и повторяем
3. **Попытки повтора**: Продолжаем до `retry` раз
4. **Успех**: Прекращаем повторы при первом успехе
5. **Финальная неудача**: После исчерпания всех попыток задача считается неудачной

### Пример вывода

```
⌛   Executing add-routes ... 
⛔   Executing add-routes failed after 2.5s
  ▪ Error: exit status 1, connection refused
  ▪ Retrying in 30s...
⌛   Executing add-routes (attempt 2/4) ...
✅   Executing add-routes (attempt 2/4) completed after 1.2s
```

## Очередь задач

Задачи выполняются последовательно в очереди FIFO (First-In-First-Out).

### Почему последовательно?

- **Предотвращение конфликтов**: Несколько задач не изменяют один роутер одновременно
- **Управление ресурсами**: Избегаем перегрузки роутера одновременными запросами
- **Предсказуемое поведение**: Задачи выполняются в детерминированном порядке

### Как это работает

```
Задача 1 срабатывает в 06:00 → Добавлена в очередь
Задача 2 срабатывает в 06:00 → Добавлена в очередь (ждет Задачу 1)
Задача 3 срабатывает в 06:01 → Добавлена в очередь (ждет Задачу 2)

Порядок выполнения: Задача 1 → Задача 2 → Задача 3
```

### Последствия

- Если задача выполняется долго, последующие задачи будут ждать
- Используйте подходящие интервалы, чтобы избежать накопления очереди
- Отслеживайте время выполнения для оптимизации расписаний

## Примеры

### Пример 1: Простое периодическое обновление

Обновление маршрутов без удаления каждые 3 часа на нескольких роутерах:

```yaml
tasks:
  - name: "Обновление маршрутов каждые 3 часа"
    commands:
      - add-routes
    configs:
      - /path/to/router1.yaml
      - /path/to/router2.yaml
      - /path/to/router3.yaml
    interval: "3h"
```

### Пример 2: Ежедневное обслуживание с повтором

Полное обновление маршрутов с удалением всех старых ежедневно в 2 часа ночи с повтором при ошибке:

```yaml
tasks:
  - name: "Ежедневное обновление маршрутов"
    commands:
      - delete-routes
      - add-routes
    configs:
      - /path/to/router.yaml
    times:
      - "02:00"
    retry: 3
    retryDelay: "1m"
```

### Пример 3: Несколько временных окон

Обновление DNS в несколько раз в течение дня:

```yaml
tasks:
  - name: "Обновление DNS"
    commands:
      - add-dns-records
    configs:
      - /path/to/router.yaml
    times:
      - "06:00"
      - "12:00"
      - "18:00"
      - "23:00"
```

### Пример 4: Параллельное выполнение для нескольких роутеров

Одновременное выполнение команд на всех роутерах для более быстрого завершения:

```yaml
tasks:
  - name: "Обновление всех роутеров параллельно"
    commands:
      - add-routes
    configs:
      - /path/to/router1.yaml
      - /path/to/router2.yaml
      - /path/to/router3.yaml
      - /path/to/router4.yaml
    interval: "3h"
    strategy: "parallel"
    retry: 2
    retryDelay: "30s"
```

### Пример 5: Сложная настройка для нескольких роутеров

```yaml
tasks:
  # Частые обновления для критичных роутеров
  - name: "Критичные роутеры - каждый час"
    commands:
      - add-routes
    configs:
      - /path/to/critical1.yaml
      - /path/to/critical2.yaml
    interval: "1h"
    retry: 2
    retryDelay: "30s"

  # Менее частые обновления для второстепенных роутеров
  - name: "Второстепенные роутеры - каждые 6 часов"
    commands:
      - add-routes
    configs:
      - /path/to/secondary1.yaml
      - /path/to/secondary2.yaml
    interval: "6h"
```

## Советы

### 1. Выбирайте подходящие интервалы

- **Слишком часто**: Тратит ресурсы, может перегрузить роутер
- **Слишком редко**: Данные могут устареть
- **Рекомендуется**: Начните с 3-6 часов, корректируйте по необходимости

### 2. Используйте retry

- Включайте retry для операций, зависящих от сети
- Устанавливайте разумный `retryDelay` (30s - 2m)
- Не превышайте 3-5 попыток повтора

### 3. Планируйте обслуживание в непиковое время

```yaml
times:
  - "02:00"  # Время низкого трафика
```

### 4. Тестируйте конфигурацию без планировщика

```bash
# Протестируйте отдельные команды перед планированием
./gokeenapi add-routes --config router.yaml

# Затем добавьте в планировщик
```

## Запуск как сервис

### systemd (Linux)

Создайте `/etc/systemd/system/gokeenapi-scheduler.service`:

```ini
[Unit]
Description=Gokeenapi Scheduler
After=network.target

[Service]
Type=simple
User=youruser
WorkingDirectory=/path/to/gokeenapi
ExecStart=/path/to/gokeenapi scheduler --config /path/to/scheduler.yaml
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

Включите и запустите:
```bash
sudo systemctl enable gokeenapi-scheduler
sudo systemctl start gokeenapi-scheduler
sudo systemctl status gokeenapi-scheduler
```

### Docker

```bash
docker run -d \
  --name gokeenapi-scheduler \
  --restart unless-stopped \
  -v /path/to/scheduler.yaml:/scheduler.yaml \
  -v /path/to/configs:/configs \
  noksa/gokeenapi:stable \
  scheduler --config /scheduler.yaml
```
